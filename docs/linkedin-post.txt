Eu não montei um "projeto de tutorial". Montei um laboratório de QA full-stack com agente de IA e menu interativo — e quase tudo deu errado antes de dar certo. São 4 problemas que me fizeram repensar como a gente aprende MCP, testes e automação de verdade.

━━━━━━━━━━━━━━━━━━━━

Problema 1: Tutorial de MCP = exemplo genérico. Eu queria um projeto real.

Decisão: construir o terreno antes do agente. Criei o QA Lab — API (Node + Express), frontend (Next.js), PostgreSQL, testes Cypress por área (auth, api, admin, ui). Só assim o agente teria algo de verdade para rodar.

━━━━━━━━━━━━━━━━━━━━

Problema 2: "Rodar teste" virou bagunça.

Rodar tudo = demora. Decorar comando de cada suíte = nada amigável. Solução: servidor MCP com "run_tests" + agente com menu. Você escolhe por número (1 = todos, 2 = Admin, 3 = Auth...) — zero comando para decorar.

━━━━━━━━━━━━━━━━━━━━

Problema 3: Reexecutar teste com os mesmos dados quebrava (409 vs 201).

Queria usar meus dados às vezes. Rodei de novo → API devolve 409 → teste que esperava 201 quebrava. Solução: asserções aceitam 201 ou 409. Reexecução deixou de ser erro.

━━━━━━━━━━━━━━━━━━━━

Tela que demora 15s passa em tudo e é péssima pro usuário. Criei o TICTAC — teste de performance no caminho crítico (carregamento, health, tempo até formulário e dashboard). Limites configuráveis. Se estourar, falha antes de "está lento" na boca do usuário.

✅ Dados de imersão opcionais + testes que aceitam 201 ou 409

I didn’t build a “tutorial project.” I built a full-stack QA lab with an AI agent and an interactive menu — and almost everything went wrong before it went right. Here are 4 problems that made me rethink how we actually learn MCP, testing, and automation.

━━━━━━━━━━━━━━━━━━━━

Problem 1: MCP tutorials = generic examples. I wanted a real project.

Decision: build the ground before the agent. I created QA Lab — API (Node + Express), frontend (Next.js), PostgreSQL, Cypress tests by area (auth, api, admin, ui). Only then would the agent have something real to run.

━━━━━━━━━━━━━━━━━━━━

Problem 2: “Run tests” became a mess.

Run everything = slow. Memorize each suite’s command = not friendly. Solution: MCP server with "run_tests" + agent with menu. You pick by number (1 = all, 2 = Admin, 3 = Auth…) — zero commands to memorize.

━━━━━━━━━━━━━━━━━━━━

Problem 3: Re-running tests with the same data broke them (409 vs 201).

I wanted to use my own data sometimes. Ran again → API returns 409 → test expecting 201 failed. Solution: assertions accept 201 or 409. Re-running is no longer an error.

━━━━━━━━━━━━━━━━━━━━

Problem 4: “It works” isn’t enough.

A screen that takes 15s can pass every test and still suck for the user. I built TICTAC — a performance test on the critical path (load time, API health, time to form and dashboard). Configurable limits. If it blows past, it fails before users say “it’s slow.”

━━━━━━━━━━━━━━━━━━━━

What’s left at the end:

✅ Real project: backend, frontend, DB, organized tests
✅ MCP + agent with menu — choose what to run, no commands to memorize
✅ Optional immersion data + tests that accept 201 or 409
✅ Performance test (TICTAC) on the critical path
✅ Docs + visual page: manual vs agent (link in comments)

Every problem became: “how do I turn this into a reusable solution?” The result is a lab to train automation, E2E, API, MCP and agents on the same codebase.
