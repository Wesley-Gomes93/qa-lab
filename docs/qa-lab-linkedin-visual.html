QA Lab – Como tudo se desenrola

Full-stack + Cypress + Playwright + Pipeline CI/CD + agente MCP

Sobre o projeto

O QA Lab é um playground full-stack para aprender e praticar automação de testes, CI/CD e agentes (MCP). Aplicação completa: frontend (Next.js), backend (Express), banco (PostgreSQL), suíte Cypress e Playwright, pipeline CI/CD (lint → build → testes → report) e agente interativo.

Você pode rodar os testes manualmente, via agente (menu MCP) ou deixar a pipeline rodar no GitHub Actions. O agente oferece menu para escolher suíte ou arquivo específico, com dados de imersão opcionais.

Frontend (Next.js)

Interface do usuário: tela inicial (Playground) com registro, login e healthcheck da API; dashboard para administradores.

Responsável por: exibir formulários, chamar a API, redirecionar após login, listar/editar/excluir usuários (admin), filtro de pesquisa e validação de idade (18–80).

Backend (Express)

API REST em Node.js: autenticação (registro, login com JWT), CRUD de usuários, validações e regras de negócio.

Responsável por: persistir usuários no PostgreSQL, validar idade 18–80, impedir exclusão do admin, retornar tokens e dados consistentes para o frontend.

PostgreSQL

Banco de dados relacional que armazena usuários e garante persistência entre reinícios do backend.

Responsável por: tabela de usuários (id, nome, email, senha hash, idade, ativo), integridade dos dados e suporte às consultas da API.

Cypress + Playwright (E2E + API)

Duas suítes de testes em paralelo: Cypress e Playwright cobrindo auth, api, admin, ui e performance. Page objects, helpers e specs organizados por cenário.

Responsável por: fluxos de registro, login, logout, healthcheck, API, dashboard admin (editar idade, validação 18-80, inativo, filtro, exclusão), UI e performance (TICTAC). Pipeline CI/CD: lint → build → testes → report.

MCP Server (run_tests)

Servidor que implementa o protocolo MCP e expõe a ferramenta run_tests para executar o Cypress com suíte ou spec específico.

Responsável por: receber chamadas do agente (suite ou spec + dados opcionais de imersão), repassar variáveis de ambiente ao Cypress e devolver o resultado da execução.

Agente (Menu)

Script Node que se conecta ao MCP, lista ferramentas e oferece um menu interativo para escolher o que rodar.

Responsável por: perguntar se o usuário quer informar dados para imersão (nome, e-mail, senha, idade), exibir opções de suíte e arquivos com descrições em português, chamar run_tests com os parâmetros escolhidos e exibir o resultado.

Resumo

Manual: você sobe cada parte (banco, backend, frontend) e roda npm test (ou um --spec) na pasta tests/.

Pelo agente: um comando (npm run agent:run-tests), menu com todas as opções de teste, você escolhe por número e o agente + MCP executam o Cypress e devolvem o resultado. Tudo integrado.

Como usar os prompts de imersão?

O projeto tem um guia em docs/prompts-imersao-testes.md. Ele não é um script que você executa no terminal: é um conjunto de prompts para colar numa conversa com uma IA (Cursor, ChatGPT, Claude, etc.).

Como “executar”: (1) Abra um chat com a IA e cole o prompt de contexto do arquivo (aquele que começa com “Você é meu assistente de QA no projeto QA Lab…”). (2) Diga qual teste quer rodar (ex.: registro, login, edição de idade). (3) A IA pergunta se você quer manual (passo a passo no site) ou automatizado (Cypress aberto para ver o navegador rodando) e se os dados você digita ou ela usa aleatórios. (4) Você responde e a IA te guia: se manual, te manda o passo a passo para abrir o site e preencher; se automatizado, te passa o comando para abrir o Cypress e te diz para clicar em Run para assistir ao teste na tela.

Resumindo: você “executa” o guia colando o prompt no chat e conversando com a IA; ela te orienta e te diz exatamente o que fazer (manual no site ou Cypress em modo aberto).
