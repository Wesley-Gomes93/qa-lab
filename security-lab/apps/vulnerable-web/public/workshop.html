🔒 Workshop — Exploração passo a passo

Use este lab para praticar. Cada vulnerabilidade tem passos e um botão "Testar" que abre o exploit no lab mesmo.

1. Reflected XSS (XSS Refletido)

O que é: O servidor reflete o que você digita na URL na página, sem sanitizar. Se for HTML/JS, executa.

Passo 1. Clique no botão abaixo para abrir a URL com payload.
Passo 2. Se um popup 1 aparecer → vulnerável.

/search?q=<script>alert(1)</script>

Testar Reflected XSS

2. Stored XSS (XSS Armazenado)

O que é: O script fica salvo no servidor. Qualquer um que abrir a página de comentários vai executá-lo.

Passo 1. Clique em "Ir para Comentários" e no campo digite:

<img src=x onerror=alert(1)>

Passo 2. Envie o formulário. Recarregue a página — o popup aparece.
Ir para Comentários

3. DOM-based XSS

O que é: O JavaScript da página lê da URL (hash #) e coloca no DOM sem sanitizar.

Passo 1. Clique no botão — a URL já traz o payload no hash.
Passo 2. O popup deve aparecer.

/dom-xss.html#<img src=x onerror=alert(1)>

Testar DOM XSS

4. Open Redirect

O que é: O app redireciona para qualquer URL que você passar. Risco de phishing.

Passo 1. Clique no botão — você será redirecionado para example.com.
Passo 2. Troque na URL por https://evil.com para simular ataque.
Testar Open Redirect

5. Information Disclosure — /debug

O que é: Endpoint que expõe dados sensíveis (senhas, env vars).

Passo 1. Abra /debug e veja o JSON com senha de admin.
Abrir /debug

6. Information Disclosure — Comentário HTML

O que é: Desenvolvedores deixam chaves/senhas em comentários HTML (Ctrl+U).

Passo 1. Abra a página inicial, pressione Ctrl+U (ou Cmd+Option+U no Mac).
Passo 2. Procure por <!-- Dev note: — há uma chave exposta.
Abrir index e ver código-fonte

7. Path Traversal

O que é: O parâmetro name aceita ../ e permite ler arquivos fora do diretório.

Passo 1. No Mac/Linux, clique abaixo para tentar ler /etc/passwd.
Passo 2. Se aparecer conteúdo do arquivo → vulnerável.

/file?name=../../../etc/passwd

Testar Path Traversal

8. CSRF (falta de token)

O que é: O formulário de comentários não tem token CSRF. Um site malicioso pode enviar POST em nome do usuário.

Passo 1. Abra o DevTools (F12) → aba Network.
Passo 2. Envie um comentário em /comments. Veja que o POST não tem header anti-CSRF.
Ir para Comentários (verificar CSRF)
